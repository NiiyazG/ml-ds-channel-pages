# ML. Урок 51/2 — Кодирование категориальных признаков
## One-hot и Label — два главных способа

Два способа, которые покрывают 90% случаев. У каждого своя логика.

### Способ 1. One-hot encoding (дословно: «один горячий»)

Самая распространённая техника. Работает так:

Допустим, у тебя столбец «город» с тремя значениями: Москва, Казань, СПб.

Вместо одного столбца «город» ты создаёшь три новых столбца:
- «город_Москва»
- «город_Казань»
- «город_СПб»

Если клиент из Москвы — в столбце «город_Москва» ставится 1, в остальных — 0.
Если из Казани — 1 в «город_Казань», 0 в остальных.

**Наглядный пример:**

Было:

| Клиент | Город |
|--------|-------|
| Иванов | Москва |
| Петров | Казань |
| Сидоров | Москва |
| Кузнецов | СПб |

Стало:

| Клиент | город_Москва | город_Казань | город_СПб |
|--------|-------------|-------------|-----------|
| Иванов | 1 | 0 | 0 |
| Петров | 0 | 1 | 0 |
| Сидоров | 1 | 0 | 0 |
| Кузнецов | 0 | 0 | 1 |

Красота в том, что теперь все числа — 0 и 1. Модель не может их умножать, делить или складывать как настоящие числа. Она просто видит: «это Москва» (первый столбец активен) или «это не Москва» (первый столбец равен 0).

**Когда one-hot хорош:**
- Категорий мало — 2, 3, 5, до 10-15 штук
- Пол: мужской/женский → 2 столбца (но обычно делают один: 1 = мужской, 0 = женский)
- Тип оплаты: наличные/карта → 2 столбца
- Цвет товара: 5 цветов → 5 столбцов

**Когда one-hot плох:**
- Категорий много. Например, 100 городов → 100 новых столбцов. Таблица становится огромной. 1 000 товаров → 1 000 столбцов. Модель не может с этим работать, это называют «проклятие размерности».
- Редкие категории. Если город Набережные Челны встречается 2 раза из 10 000 — столбец почти всегда будет нулевым. Бесполезная колонка.

### Способ 2. Label encoding (кодирование метками)

Просто нумеруешь категории: Москва = 1, Казань = 2, СПб = 3.

Всё. Один столбец, никаких 100 новых колонок.

Пример:

| Клиент | Город | Код города |
|--------|-------|-----------|
| Иванов | Москва | 1 |
| Петров | Казань | 2 |
| Сидоров | Москва | 1 |
| Кузнецов | СПб | 3 |

**Почему это может быть опасно:**

Модель видит числа и думает, что это настоящие числа. Она честно считает:
- СПб (3) в 3 раза больше Москвы (1)
- Среднее арифметическое между Москвой (1) и СПб (3) = Казань (2)

Но СПб — не «3 Москвы». Казань — не среднее между двумя столицами. Модель делает бессмысленные вычисления.

**Когда Label можно использовать:**

**Ситуация А.** Категории имеют порядок.

Оценки: отлично, хорошо, удовлетворительно.

Здесь есть естественный порядок: отлично > хорошо > удовлетворительно. Label encoding (отлично = 3, хорошо = 2, удовл. = 1) даёт модели правильную информацию: чем выше число, тем лучше оценка.

**Ситуация Б.** Ты используешь деревья решений (Random Forest, XGBoost, Gradient Boosting).

Деревья решений не оперируют числами как линейными. Они просто делят данные: «если признак ≤ 2, то иди налево, если > 2 — направо». Для них числа 1, 2, 3 — просто метки-разделители, а не «3 больше 1».

Поэтому для деревьев Label encoding безопасен даже для номинальных признаков.

**Когда Label нельзя:**

Для линейных моделей (линейная регрессия, логистическая регрессия, нейронные сети). Они воспринимают числа буквально. СПб = 3 для них означает «в три раза больше Москвы».

### Сравнение двух способов

| | One-hot | Label |
|--|---------|-------|
| Сколько столбцов | Столько же, сколько категорий | Один |
| Честность перед моделью | Честно (не создаёт ложных связей) | Может врать о порядке |
| Когда подходит | Мало категорий, линейные модели | Много категорий, деревья решений |
| Когда не подходит | Сотни категорий | Номинальные признаки + линейные модели |
| Пример | Пол, тип оплаты, цвет (≤15) | Оценки, 100 городов для дерева |

### Практический совет

Для номинальных признаков с небольшим числом значений (до 10-15) — используй **One-hot**. Для порядковых или для деревьев — **Label**. При сомнении между двумя — One-hot безопаснее.
